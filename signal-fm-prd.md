## 1. 产品概述

> *Signal FM 是一个面向垂直领域知识工作者的实时 AI 广播系统。它持续监控目标领域的全网信息流，由 LLM 筛选出真正有价值的内容，实时生成播报稿并合成为音频，以 FM 电台的方式持续播出——无需手动触发，永远在线，永远只播最值得听的那条。*

在信息过载的时代，知识工作者每天花费大量时间在信息筛选上。现有解决方案要么是"被动推送"（需要主动打开），要么是"静态生成"（需要手动上传触发）。没有任何产品做到了真正意义上的**持续自动播出、实时感知、领域专注**的 AI 电台体验。

Signal FM 填补这个空白：它是一个永不停歇的 AI 编辑，将全网噪声压缩成一条精选信息流，用声音送到你耳朵里。

> **"不是把内容做成播客，而是把电台这个形态，用 AI 重新发明一遍。"**

### 1.1 核心价值主张

| 场景 | 价值描述 |
|------|---------|
| **For 个人用户** | 开车、健身、做家务时，背景中持续播出你所在领域今天最重要的信息，不用看屏幕，不用手动操作。 |
| **For 企业团队** | 为销售、投研、运营团队订制专属频道，竞品动态、政策变化、行业数据自动播报，团队信息对齐成本为零。 |
| **核心差异** | 实时性 × 自动性 × 垂直性。不是"生成播客"，是"永久在线的领域电台"。 |
| **技术底座** | GLM 系列驱动评分 + 生成，CosyVoice 音色合成，全链路国产化，可私有化部署。 |

## 2. 市场背景与机会

> *音频内容消费正在经历由 AI 驱动的结构性转变。全球在线音频市场规模预计 2026 年达到 380 亿美元，AI 生成音频内容正快速崛起并重塑产品形态。*

### 2.1 三个正在发生的结构性变化

#### 变化一：音频消费从"娱乐"向"效率"迁移

播客最初以娱乐、访谈为主。2024 年以来，"每日简报"类、"领域速报"类内容增速超过传统访谈类。用户不再只在闲暇时听播客——他们希望在工作场景的碎片时间吸收专业信息。

#### 变化二：AI 生成音频从"工具辅助"走向"全自动生成"

NotebookLM 的 Audio Overview 功能让数以百万计的用户体验到"上传文件 → 自动生成对话播客"。根据 Google 官方数据，人们已累计生成超过 **350 年**时长的 Audio Overview。但这类产品仍需用户主动上传触发，没有做到"持续自动"。

#### 变化三：企业对"信息对齐"的需求从未如此迫切

在快速变化的行业中（AI、金融、跨境电商、制造），团队信息不对称是效率杀手。传统解决方案是内部 Newsletter、周会同步——效率低、覆盖滞后。一个能实时播报竞品和行业动态的"企业专属电台"是未被满足的真实需求。

### 2.2 中国市场的特殊机会

在中国，喜马拉雅虽然月活超过 **2.68 亿**，但其 AI 化程度仍停留在 TTS 朗读层面，没有真正的 AI 编辑和价值过滤能力。国产 LLM（GLM、Qwen 系列）的快速成熟使得打造全中文、可私有化的领域电台在技术上完全可行，且成本极低。

## 3. 竞品全景分析

> *通过系统调研，全球现有产品可分为四个象限：静态 vs 实时、通用 vs 垂直。Signal FM 的目标是唯一占据"实时 × 垂直"象限的产品。*

### 3.1 竞品对比矩阵

| 产品 | 核心模式 | 核心能力 | 致命弱点 | 定位 |
|------|---------|---------|---------|------|
| **Google NotebookLM** | 用户上传文档 → AI 双主播对话生成 | 音质最佳 / Google 生态 / 免费 | ❌ 必须手动上传内容<br>❌ 无实时抓取能力<br>❌ 无价值过滤<br>❌ 生成一次即结束 | 个人研究助手 |
| **ElevenLabs GenFM** | 用户粘贴 URL → AI 双主播播客 | 音质极佳 / 32 语言 / 秒级生成 | ❌ 完全依赖手动触发<br>❌ 每次处理单一来源<br>❌ 无持续监控能力 | 内容消费工具 |
| **华盛顿邮报 Personal Podcast** | 基于阅读历史自动生成日报播客 | 个性化 / 媒体品牌背书 | ❌ 仅限自有内容<br>❌ 幻觉问题已曝光<br>❌ 不开放 / 非实时 | 媒体订阅产品 |
| **麦悠电台（中国）** | 用户配置 RSS → TTS 朗读 / AI 改写 | 中文优化 / RSS 支持 / 本地 TTS | ❌ 无 AI 价值评分过滤<br>❌ 全量朗读无选择<br>❌ 无企业级能力 | 个人效率工具 |
| **喜马拉雅 AI 主播** | 内容平台 + TTS 朗读 + AIGC 故事 | 用户规模 2.68 亿 / 版权内容 | ❌ TTS 朗读为主无 AI 编辑<br>❌ 无实时新闻<br>❌ 通用娱乐定位 | 泛娱乐音频平台 |
| **BBC My Club Daily** | AI 生成球队专属播客 | 垂直聚焦 / 媒体品牌 | ❌ 单一垂直（体育）<br>❌ 非实时 / 无通用框架 | 媒体功能实验 |
| **▶ Signal FM（目标）** | 全自动监控 → AI 筛选 → 实时播出 | ✅ 持续实时<br>✅ AI 价值评分<br>✅ 垂直可配<br>✅ 企业私有化 | 唯一永不停歇自动 AI 电台<br>唯一内置价值过滤引擎 | **垂直实时 AI 电台** |

### 3.2 竞品共同的结构性弱点

- 所有竞品都需要用户手动触发内容输入，没有自主实时监控能力
- 没有任何竞品内置内容价值评分机制，全量处理导致信噪比低
- 无产品以"持续广播"为核心体验，均为"一次生成一次消费"模式
- 国内产品（麦悠电台）虽接入 RSS，但缺乏 AI 编辑层，本质是 TTS 朗读工具

## 4. 竞争定位与差异化战略

> *所有竞品都在做"把内容变成播客"。Signal FM 做的是"把电台这个持续广播的形态，用 AI 重新实现"——这是本质不同的产品哲学。*

### 4.1 四维差异化矩阵

| 维度 | 竞品现状 | Signal FM |
|------|---------|-----------|
| **持续性** | 触发式，用完就结束 | 永续运行。就像真正的电台——没有人在打开它之前它就已经在播了 |
| **智能筛选** | 朗读/改写所有内容，无质量过滤 | LLM 对每条内容评分，只有高价值内容才能"上播"，过滤掉 80% 噪声 |
| **实时抓取** | 依赖用户手动提供内容 | 自主爬取 RSS、API、网页，无需用户参与，内容自动流入 |
| **企业级** | 面向个人，无私有化方案 | 支持私有化部署，B2B 订制频道，适合金融/制造/投研敏感行业 |

### 4.2 目标用户画像

#### 主力用户 A · 高强度信息消费者

**画像**：投研分析师、AI 产品经理、跨境电商运营、科技记者

**特征**：每天需要追踪某个领域 20+ 个来源，信息密度极高，时间极度稀缺

**场景**：通勤、运动、吃饭时用耳朵"刷"领域资讯

#### 主力用户 B · 企业知识团队

**画像**：销售情报团队、竞品分析团队、投资机构研究部门

**特征**：需要全团队对竞品动态、政策变化保持实时认知同步

**场景**：办公室"背景广播"，会议室开场情报简报

#### 次级用户 C · 内容创作者

**画像**：Newsletter 作者、播客主持人、行业 KOL

**用途**：用 Signal FM 替代人工 RSS 监控，快速发现值得深度报道的选题

## 5. 核心功能规格

### 5.1 功能模块清单

| # | 功能模块 | 描述 | 优先级 |
|---|---------|------|--------|
| 1 | **频道编辑器** | 可视化配置数据源（RSS/API/关键词爬虫）、领域关键词、过滤规则、播报语气风格。支持多频道并行，每个频道独立运行 | P0 |
| 2 | **智能评分引擎** | 基于 GLM-4-Flash 的四维评分系统（新颖性/重要性/领域匹配/时效性），阈值可调。同类内容自动去重，防止洗稿内容入队 | P0 |
| 3 | **AI 播报脚本生成** | 用 GLM-4 将原文改写为主播口语稿，支持多种播报风格（严肃简报/幽默点评/深度解读）。强制插入来源核查标记，降低幻觉风险 | P0 |
| 4 | **多音色 TTS 合成** | CosyVoice 本地合成 + 火山引擎/Edge TTS 云端备份。支持自定义音色克隆（企业版），多语言（中/英/粤）切换 | P0 |
| 5 | **播放队列调度** | 按评分 × 时效性动态排队，高分内容优先插播。支持"插播模式"——重大突发事件立即打断当前播报 | P0 |
| 6 | **多端接入** | Web 播放器 / HLS 推流（智能音箱）/ RSS 音频订阅 / 飞书/企微 Bot 推送（含文字摘要） | P1 |
| 7 | **对话式追问** | 听完某条新闻后，通过语音或文字追问背景和影响，AI 实时回答 | P2（Roadmap Q3）|
| 8 | **企业私有化部署** | 完整 Docker 化，支持内网部署。内部知识库接入（企业 Wiki/飞书文档）。合规审计日志 | P1（企业版）|

### 5.2 差异化亮点功能详述

#### 内容价值评分系统（核心护城河）

每条进入系统的内容都会经过 LLM 的四维评估，评分低于阈值的内容直接丢弃，不进入播放队列。

| 评分维度 | 权重 | 评估标准 |
|---------|------|---------|
| 新颖性 Novelty | 30 分 | 信息是否为首次报道或独家 |
| 重要性 Importance | 30 分 | 对领域影响程度和深度 |
| 领域匹配 Relevance | 20 分 | 与订阅频道主题的相关度 |
| 时效紧迫 Urgency | 20 分 | 内容时效性和紧迫程度 |
| **入队阈值** | **合计 ≥ 70** | **低于阈值直接丢弃** |

评分标准由用户通过自然语言描述定义，例如："优先播报涉及 GPT-5/Claude 的突破性进展，淡化纯商业融资新闻。"

#### 突发插播机制

系统持续计算"紧急度评分"。当一条新内容紧急度超过阈值，且当前播报内容紧急度较低时，系统在当前句子结束时自动切换，播报紧急内容后回到原队列。这是传统播客工具完全不具备的实时响应能力。

#### 幻觉防控机制

生成脚本时强制注入信源标记（如"据 The Verge 报道"），禁止 LLM 补全未在原文出现的数据和引用。专项应对华盛顿邮报 AI 播客暴露的核心问题（幻觉和错误引用）。

## 6. 技术架构

### 6.1 五层流水线架构

```
┌─────────────────── SIGNAL FM · SYSTEM ARCHITECTURE ───────────────────┐

[ LAYER 1 · DATA INGESTION ]
  RSS Feeds ──┐
  NewsAPI    ─┼──► Dispatcher ──► Redis Stream ──► URL Dedup Hash
  Playwright ─┘  (APScheduler)   (Message Queue)   (SHA256)

[ LAYER 2 · AI INTELLIGENCE ]
  Redis Stream ──► Score Worker              Score Formula:
                   │  GLM-4-Flash            novelty(30) + importance(30)
                   │  ~100ms/item            + relevance(20) + urgency(20)
                   │                         ─────────────────────────────
                   ├─[score ≥ 70]──► Script Generator (GLM-4)
                   │                  │  口语化改写 + SSML 标注
                   │                  │  幻觉防控注入
                   └─[score < 70]──► DISCARD

[ LAYER 3 · AUDIO SYNTHESIS ]
  Script ──► TTS Router
              ├── CosyVoice (本地, 低延迟)
              └── 火山 TTS  (云端备份)
              ──► Audio Cache (MinIO)

[ LAYER 4 · BROADCAST ENGINE ]
  Audio Queue ──► Priority Scheduler
                   │  score × freshness 排序
                   │  突发插播检测
                   ▼
              FFmpeg Mixer ──► HLS Stream ──► Web Player
                           └──► RTMP ──────► Smart Speaker
                           └──► RSS ───────► Podcast App

[ LAYER 5 · INTERFACE ]
  WebSocket ──► Real-time Metadata (title / score / source)
  REST API  ──► 频道配置 / 播放控制 / 数据看板
  Bot Push  ──► Feishu / WeCom (文字摘要 + 评分)

└────────────────────────────────────────────────────────────────────────┘
```

### 6.2 技术选型

| 组件 | 推荐方案 | 备选 | 说明 |
|------|---------|------|------|
| 抓取调度 | APScheduler + Celery | Airflow | 按源设不同频率 |
| 消息队列 | Redis Stream | Kafka | 轻量，支持 Consumer Group |
| LLM 评分 | GLM-4-Flash | Qwen-Turbo | 每条约 ¥0.001，极低成本 |
| 脚本生成 | GLM-4 | Claude Sonnet | 中文口语化效果最佳 |
| TTS | CosyVoice 2.0 | Edge TTS / 火山 | 开源，支持音色克隆 |
| 推流 | FFmpeg → Icecast | SRS | 输出标准 HLS / MP3 流 |
| 音频缓存 | MinIO | 阿里 OSS | 私有化可本地部署 |
| 前端播放器 | Web（WebSocket） | — | 实时元数据推送 |

### 6.3 关键技术决策说明

**为何用 GLM-4-Flash 做评分而不是规则过滤？**

规则过滤无法理解"信息价值"的语义——"OpenAI 发布 GPT-5"和"某 AI 公司发布大模型"在关键词层面很难区分，但前者信息价值远高于后者。LLM 评分成本极低（每条约 ¥0.001），但质量远超关键词匹配。

**为何选择 CosyVoice？**

CosyVoice 是达摩院开源的 TTS 模型，在中文语音自然度上显著优于 Edge TTS，且支持音色克隆，是企业版"专属主播"功能的基础。本地部署可消除云端 API 的延迟抖动。

## 7. 竞争壁垒与护城河

### 7.1 六大护城河

**① 数据飞轮**
用户对播报内容的反馈（跳过/追问/收藏）持续优化评分模型。随着使用时间增长，系统对该用户领域的"价值判断"越来越准，形成个人化护城河。

**② 频道知识图谱积累**
系统长期追踪某个领域，会建立该领域的实体关系图（公司、人物、技术术语），使得去重、关联播报、上下文理解能力持续提升。这是新进入者短期内无法复制的。

**③ 企业私有化壁垒**
企业部署后，内部知识库、历史播报记录、音色库都在客户侧。迁移成本极高，形成强黏性。

**④ GLM 生态绑定**
与智谱 AI 的 GLM 系列深度整合，在模型更新、API 优先级、私有化支持上具备其他产品不具备的优势。企业采购 AI 基础设施时，Signal FM 可作为增值能力打包销售。

**⑤ 评分系统可解释性设计**
用户可以看到每条内容为什么被播报、每个维度的得分，建立信任感。这是 NotebookLM 和 GenFM 完全不具备的透明度。

**⑥ 先发的品类定义权**
在"垂直领域实时 AI 电台"这个品类，目前没有成熟竞品。先发者有机会定义品类认知，形成用户心智中的标杆。时间窗口估计为 **12-18 个月**。

## 8. 商业模式

Signal FM 采用 **B2C 订阅 + B2B 企业授权**的双轨模式，以 B2C 建立品牌认知，以 B2B 实现规模盈利。

### 8.1 定价分层

| | Free | Pro ⭐ | Enterprise |
|--|------|--------|-----------|
| **价格** | ¥0 / 月 | ¥49 / 月 | 定制 / 年 |
| 频道数量 | 1 个 | 5 个 | 无限 |
| 数据源配置 | 预设模板 | 完全自定义 | 完全自定义 |
| 播出时长 | 每日 30 分钟 | 无限 | 无限 |
| TTS 音色 | 标准 | 多音色切换 | 音色克隆 |
| Bot 推送 | ❌ | ✅ 飞书/企微 | ✅ 飞书/企微 |
| 评分自定义 | ❌ | ✅ | ✅ |
| 历史回放 | 7 天 | 30 天 + 导出 | 无限 |
| 私有化部署 | ❌ | ❌ | ✅ Docker |
| 知识库接入 | ❌ | ❌ | ✅ |
| SSO / RBAC | ❌ | ❌ | ✅ |
| SLA / 专属支持 | ❌ | ❌ | ✅ |
| API 开放 | ❌ | ❌ | ✅ |

### 8.2 B2B 场景策略

**金融 / 投研**
竞争对手动态、政策监测频道，帮助研究员在晨会前完成信息同步。定价锚定信息终端（Bloomberg 等），具备明显价格优势。

**制造 / 跨境电商**
供应链风险、关税政策、竞品发布频道。与智谱 AI OpenClaw 体系深度整合，作为 Agent 的"情报广播"层。

**企业内训 / 知识管理**
将企业内部文档（周报、会议纪要、知识库）转化为员工可收听的内部电台，降低知识传递摩擦。
